## 1. 프로젝트 개요

배트맨 승무패는 매주 발행되는 회차마다 국내외 축구 14경기의 승/무/패를 예측하는 게임이다. 본 프로젝트의 목표는 회차가 발행되면 서버에서 대상 14경기를 자동으로 수집하고, 각 경기에 대한 최신 부가 정보(팀 폼, 상대 전적, 순위, 결장자, 배당 흐름 등)를 결합하여, 발매 마감 12시간 전 시점에 로컬 CLI 환경의 Claude AI로 경기별 상세 분석과 회차 종합 리포트를 자동 생성하는 것이다.

핵심 요구사항은 세 가지로 정리된다. 첫째, 회차 정보는 매주 서버에서 자동으로 크롤링한다. 둘째, 경기 정보는 최신성이 최우선이므로 분석 시점은 회차 발매 마감 12시간 전으로 고정한다(결장자·배당 변동을 최대한 반영하면서도 분석·검토 여유를 확보하는 시점). 셋째, 분석 자체는 로컬 CLI에서 Claude를 호출하여 수행한다.

벤치마크 대상은 zentoto.com의 축구 토토 페이지로, 회차별 경기 목록, 국내/해외 배당 비교, 경기별 통계·전적 제공 방식을 참고한다.

## 2. 대상 서비스 이해

### 2.1 배트맨 승무패 구조

승무패(게임 ID `G011`)는 회차 단위로 발행되며, 한 회차는 축구 14경기로 구성된다. 배트맨 URL 구조에서 `gmTs` 파라미터가 회차 식별자다. `gmTs=260038` = "2026년 38회차" 규칙은 1주차 PoC에서 응답 필드(`gmOsidTsYear`/`gmOsidTs`)로 **확증**되었다.

수집해야 할 회차 단위 정보는 다음과 같다: 회차 번호, 발매 시작/마감 일시, 경기 목록(14경기), 각 경기의 홈/원정 팀명, 리그, 경기 일시, **승/무/패 대중 투표 분포**, 핸디캡 여부, 경기 상태(발매중/마감/취소).

**중요(PoC 확인):** 승무패는 프로토(승부식)와 달리 패리뮤추얼 방식이라 **배트맨 자체 고정 배당이 존재하지 않는다**(배당 필드가 전부 0). 대신 경기별 승/무/패 투표 수가 제공되며, 본 시스템은 이 대중 투표 분포를 "배트맨 배당"의 대체 신호로 사용한다. 대중 쏠림과 해외 시장 확률의 괴리가 이변 탐지의 핵심 지표가 된다. 상세는 `docs/poc-betman.md`.

### 2.2 벤치마크: zentoto.com 페이지 분석

zentoto는 회차별 14경기를 표 형태로 정리하고, 경기마다 배트맨 배당과 해외 배당을 나란히 비교해 보여주는 것이 핵심 가치다. 본 프로젝트에서는 zentoto를 단순 참고가 아니라 **정식 분석 대상**으로 삼아, 1주차 PoC 기간에 아래 관점으로 페이지 구조와 정보 설계를 분석하고 그 결과를 리포트 설계에 반영한다. 참고로 zentoto 역시 자동 접근이 제한되는 사이트이므로, 벤치마크 분석은 브라우저 기반 수동 분석(화면 캡처 + 정보 구조 기록)으로 수행하며, zentoto 데이터를 크롤링하여 사용하는 것은 범위에서 제외한다(동일 정보는 4.3절의 공식 API 소스로 대체).

| 분석 관점 | 확인 항목 | 우리 시스템 반영 |
|---|---|---|
| 회차 정보 구조 | 회차 선택 UI, 마감시각 표기, 경기 수/상태 표시 방식 | 회차 종합 리포트 상단 메타 영역 설계 |
| 경기 목록 테이블 | 컬럼 구성(경기시각, 리그, 팀, 배당, 픽), 정렬 기준, 모바일 대응 | 14경기 요약 표의 컬럼·정렬 설계 |
| 배당 비교 방식 | 배트맨 배당 vs 해외 배당 병렬 표기, 내재확률 환산 표시 여부, 배당 변동 표시 | "투표 분포–시장 확률 괴리" 지표의 계산·표기 방식 |
| 경기 상세 정보 | 최근 폼, 상대 전적, 순위, 홈/원정 분리 통계의 깊이와 표현 | 경기별 리포트의 데이터 섹션 구성 |
| 결과·검증 | 회차 종료 후 적중 표시, 과거 회차 아카이브 방식 | 7.4절 적중률 검증 루프의 결과 화면 |
| 부족한 점 (차별화 기회) | 결장자 정보 부재 여부, 분석 서술 부재(수치 나열 위주), AI 해설 없음 | **서술형 AI 분석 + 신뢰도 등급**이 본 시스템의 차별점 |

분석 산출물은 `docs/benchmark-zentoto.md`(구조 분석 노트 + 캡처)로 저장하고, 리포트 화면 설계(별첨 디자인 가이드)의 입력으로 사용한다. 해외 배당 자체는 배당 대비 기대값 판단의 핵심 신호이므로 zentoto가 아닌 별도 공식 소스에서 수집한다(4.3절).

## 3. 시스템 아키텍처

시스템은 서버(수집·스케줄링)와 로컬(AI 분석) 두 영역으로 나뉜다.

```
┌───────────────────── 서버 (VPS / 홈서버 / GitHub Actions) ─────────────────────┐
│                                                                                │
│  [1] 회차 감지기 (매일 09:00 폴링)                                              │
│        └─ 신규 회차 발견 → 회차 메타 + 14경기 목록 저장 (rounds/2026-38.json)     │
│                                                                                │
│  [2] 스케줄러                                                                   │
│        └─ 발매마감시각 파싱 → "마감 - 12h" 에 최종 수집 잡 예약                    │
│                                                                                │
│  [3] 최종 수집기 (마감 12시간 전 실행)                                           │
│        └─ 배트맨 최신 배당 재수집 + 해외배당/폼/전적/결장자 수집                    │
│        └─ 경기별 분석 입력 패키지 생성 (matches/2026-38/M01.json ~ M14.json)      │
│        └─ 로컬로 전달 (Git push → 로컬이 폴링으로 수신)                           │
└────────────────────────────────────────────────────────────────────────────────┘
                                      │
                                      ▼
┌───────────────────────────── 로컬 (분석 PC) ───────────────────────────────────┐
│  [4] AI 분석 CLI (Claude)                                                       │
│        └─ 경기별 입력 JSON → 프롬프트 템플릿 → Claude 호출 → 경기별 리포트 14개    │
│        └─ 회차 종합 리포트 1개 (전체 요약, 픽 조합, 신뢰도 순위)                   │
│  [5] 산출물: reports/2026-38/ (마크다운) → site/ 정적 페이지 빌드                 │
│        └─ git push → Cloudflare Pages 자동 배포, 알림(텔레그램·디스코드 선택)      │
└────────────────────────────────────────────────────────────────────────────────┘
```

서버와 로컬 간 데이터 전달은 프라이빗 Git 저장소를 권장한다. 서버가 수집 결과를 커밋·푸시하면 로컬 CLI가 pull 후 분석을 수행하는 구조로, 별도 인프라 없이 이력 관리와 전달을 동시에 해결한다.

로컬 실행 트리거는 **폴링 방식으로 확정**한다. 로컬 PC(Windows)의 작업 스케줄러가 5~10분 간격으로 `git fetch`를 수행하고, 아직 리포트가 생성되지 않은 회차의 입력 패키지가 감지되면 `analyze` CLI를 자동 실행한다. 로컬에 웹훅 수신 서버를 둘 필요가 없고, PC가 꺼져 있었더라도 부팅 후 첫 폴링에서 자연스럽게 따라잡는다. 다만 마감 전까지 PC가 켜지지 않으면 분석이 누락되므로, 서버 측 감시 잡(5절의 마감−11h)이 리포트 커밋 부재를 확인해 텔레그램으로 알리고, 이때는 수동으로 `analyze --round …`를 실행하는 절차를 운영 수칙에 둔다.

## 4. 크롤링 설계

### 4.1 수집 방법 비교 및 권장안

배트맨 페이지는 JavaScript로 데이터를 렌더링하는 구조이며, 단순 HTML 요청으로는 경기 목록을 얻기 어렵다. 또한 robots 정책상 자동 접근이 제한되는 사이트이므로 접근 방식 선택이 중요하다(4.4절의 유의사항 참조).

| 방식 | 설명 | 장점 | 단점 | 판정 |
|---|---|---|---|---|
| A. 내부 JSON API 직접 호출 | 브라우저 개발자도구(Network 탭)로 페이지가 호출하는 XHR/POST 엔드포인트를 식별해 동일 요청 재현 | 가장 가볍고 빠름, 파싱 불필요 | 엔드포인트·파라미터가 비공개라 변경에 취약, 세션/헤더 요구 가능 | **1순위** |
| B. Playwright 헤드리스 렌더링 | 실제 브라우저로 페이지를 열고 렌더링된 DOM 또는 네트워크 응답을 가로채 수집 | 구조 변경에 강함, A의 엔드포인트 발견 도구로도 활용 | 무겁고 서버 리소스 필요 | **폴백 + 초기 분석 도구** |
| C. 수동 입력 백업 | 크롤링 실패 시 회차 정보를 수동 JSON으로 입력 | 항상 동작 | 자동화 아님 | 비상용 |

**역공학은 1주차 PoC에서 완료되었다.** 회차 감지는 `POST /buyPsblGame/inqBuyAbleGameInfoList.do`, 회차 상세(14경기 + 투표 분포)는 `POST /buyPsblGame/totoGameData.do`(폴백: `gameInfoInq.do`)로 확정되었고, 요청 형식·응답 스키마는 `docs/poc-betman.md`에 기록했다. 남은 검증은 httpx 직접 호출 시 세션 쿠키 필요 여부(2주차)다. 운영 단계에서는 A 방식(직접 호출)을 기본으로 하고, 응답 스키마 검증에 실패하면 자동으로 B 방식(Playwright)으로 폴백한다. 이 이중화가 "좋은 방법"의 핵심이다. 사이트 개편 시에도 B가 살아 있으므로 서비스 중단 없이 A를 재정비할 수 있다.

### 4.2 회차 감지 로직

매일 오전 1회(예: 09:00 KST) 발매 가능 게임 목록을 조회하여 `G011`의 신규 `gmTs`를 감지한다. 이미 저장된 회차와 비교해 새 회차가 있으면 즉시 14경기 목록과 발매 마감 일시를 저장하고, 스케줄러에 "마감 - 12시간" 최종 수집 잡을 등록한다. `gmTs` 증가 규칙(연도+회차)을 이용해 다음 회차를 선제 조회하는 보조 로직도 두되, 목록 페이지 기반 감지를 정본으로 삼는다.

### 4.3 부가 데이터 수집 (분석 품질의 핵심)

배트맨 자체 정보만으로는 분석 깊이가 부족하므로, 최종 수집 단계에서 경기별로 다음을 병합한다.

| 데이터 | 소스 후보 | 용도 |
|---|---|---|
| 해외 배당 (1X2, 이동 추이) | The Odds API, football-data.org, OddsPortal 등 | 시장 확률 기준선, 대중 투표 분포와의 괴리 탐지 |
| 최근 폼 (최근 5~10경기) | FBref, football-data.org, API-Football | 팀 컨디션 평가 |
| 상대 전적 (H2H) | 동일 | 매치업 특성 |
| 리그 순위·득실 | 동일 | 동기 부여(강등권/우승 경쟁) 판단 |
| 결장·부상·출전정지 | API-Football, 뉴스 검색 | 마감 12시간 전 최신화가 가장 중요한 항목 |
| 일정 피로도 (직전/직후 일정, 컵대회) | 일정 데이터 | 로테이션 가능성 |

소스는 3주차 실측을 거쳐 다음과 같이 확정되었다.

| 소스 | 실측 결과 | 확정 용도 |
|---|---|---|
| The Odds API (월 500 크레딧) | **동작 확인.** K리그1·노르웨이 엘리테세리엔 커버, **K리그2 미커버**(2026-38 기준 8/14경기 커버) | **해외 배당 정본.** 스포츠당 1크레딧, 회차당 수집 2회 ≈ 4~8크레딧. 미커버 경기는 배당 결측으로 표기하고 분석에서 명시 |
| ~~API-Football~~ (무료) | **사용 불가.** 무료 플랜은 2022~2024 시즌만 접근 가능 — 현재 시즌 폼·전적·순위·결장자 조회 불가 | 유료 플랜(월 $19~) 전환 시 재도입 옵션. `collector/enrich.py`에 병합 지점 유지 |
| **Claude CLI 웹 리서치** | 분석기가 Claude Code 헤드리스로 실행되므로 WebSearch 도구 사용 가능 | **폼·상대전적·순위·결장자·일정 피로도 정본.** 입력 패키지의 `research_needed` 목록을 분석 1단계에서 웹 검색으로 조사 — 마감 12시간 전 최신성에 유리, 추가 비용 없음. 확인 못 한 항목은 결측으로 명시 |
| football-data.org | 유럽 주요 리그만 커버 — K리그 없음 | 사용 안 함 (유럽 리그 회차에서만 폴백 후보) |

회차당 호출 수가 무료 쿼터의 예산이므로, 수집기마다 호출 카운터(응답 헤더의 잔여 크레딧)를 로그로 남겨 예산 초과 시 경고한다.

팀명 매핑이 실무상 가장 번거로운 문제다. 배트맨의 한글 팀명("맨체스터U")과 해외 API의 영문 팀명("Manchester United")을 잇는 매핑 테이블(`teams.yaml`)을 구축하고, 미매핑 팀 발견 시 경고를 남겨 수동 보완하는 운영 절차를 둔다.

### 4.4 크롤링 유의사항 (법적·윤리적)

배트맨은 robots 정책으로 자동 접근을 제한하는 사이트다. 따라서 본 시스템은 개인 연구·참고 목적의 저빈도 수집으로 범위를 한정해야 한다. 회차당 요청 횟수를 최소화하고(감지 1회/일 + 최종 수집 1회/회차 수준), 요청 간 지연을 두며, 수집 데이터를 외부에 재배포하지 않는 것을 원칙으로 한다. 서비스화(불특정 다수 제공)를 고려할 경우 사이트 이용약관 검토와 공식 데이터 제휴 여부 확인이 선행되어야 하며, 이는 본 기획 범위 밖의 별도 의사결정 사항으로 남긴다. 분석 결과물에는 "본 자료는 통계적 참고용이며 구매 결과를 보장하지 않는다"는 고지를 항상 포함한다.

## 5. 스케줄링 설계

핵심은 "마감 12시간 전"이라는 상대 시각 트리거다. 회차마다 마감 시각이 다르므로 고정 cron만으로는 부족하고, 회차 감지 시점에 동적으로 잡을 등록하는 구조가 필요하다.

| 시점 | 잡 | 내용 |
|---|---|---|
| 매일 09:00 | 회차 감지 | 신규 회차 확인, 발견 시 경기목록 저장 + "예정" 페이지 빌드·배포(7.5절 ①) + 아래 잡 동적 등록 |
| 마감 − 24h | 사전 점검 (선택) | 데이터 소스 정상 여부, 팀 매핑 누락 점검, 문제 시 알림 |
| 마감 − 12h | **최종 수집 + 분석 트리거** | 배당·결장자 최신화 → 입력 패키지 생성 → Git push → 로컬 분석 실행 |
| 마감 − 11h | 미완료 감시 | 리포트 미생성 시 재시도 및 알림 |
| 회차 종료 익일 | 결과 수집 | 실제 결과 조회 → results 기록(6.2절) → 결과 페이지 갱신(7.5절 ③) |

구현은 Python `APScheduler` 단일 프로세스를 사용한다. 잡 영속화는 별도 DB 대신 **재시작 시 `data/rounds/*.json`에서 미래 분석 시점의 잡을 재등록하는 파일 기반 복원**으로 처리한다(모든 동적 잡이 회차 파일에서 파생되므로 이 방식으로 충분하며, 회차 파생이 아닌 잡이 늘어나면 SQLite 잡 스토어로 전환한다). GitHub Actions로 운영할 경우 `schedule` 워크플로(매일 감지)와 `workflow_dispatch`/`repository_dispatch`(마감-12h 시각을 감지 단계에서 계산해 예약 이슈로 등록)를 조합할 수 있으나, 정확한 상대 시각 실행에는 자체 서버 + APScheduler가 가장 단순하고 신뢰성이 높다. **권장: 소형 VPS 1대 + APScheduler.**

모든 시각 처리는 KST(Asia/Seoul) 기준으로 통일하고, 서버 UTC 환경에서의 변환 오류를 막기 위해 타임존을 명시적으로 다룬다.

### 6.1 스키마 검증과 팀명 매핑

모든 수집 응답은 pydantic 모델로 검증한다(필수 필드 존재, 경기 수 = 14, 배당 값 범위). 검증 실패가 4.1절 Playwright 폴백과 알림을 트리거하는 단일 관문이 되도록 수집기 출력을 이 모델로 통일한다.

팀명 매핑 테이블(`config/teams.yaml`)은 배트맨 한글 팀명을 표준 영문명과 외부 API ID로 잇는다.

```yaml
# config/teams.yaml
맨체스터U: { canonical: Manchester United, api_football_id: 33 }
첼시:      { canonical: Chelsea,           api_football_id: 49 }
전북현대:  { canonical: Jeonbuk Hyundai Motors, api_football_id: 2762 }
```

### 6.2 결과·적중률 기록

회차 종료 후 결과와 예측 지표를 함께 기록한다. 7.4절 검증 루프가 이 파일을 소비한다.

```json
// results/2026-38.json
{
  "round_id": "260038",
  "settled_at": "2026-07-13T09:00:00+09:00",
  "matches": [
    {
      "match_no": 3,
      "result": "win",
      "score": "2-1",
      "predicted": {"win": 0.48, "draw": 0.27, "lose": 0.25},
      "pick": "win",
      "pick_correct": true,
      "brier": 0.4058
    }
  ],
  "round_metrics": {"hit_rate": 0.571, "mean_brier": 0.58, "log_loss": 0.98}
}
```

`result`는 홈 기준 `win/draw/lose`이며, 취소·연기 경기는 `void`로 기록하고 지표 계산에서 제외한다.

### 6.3 경기 예외 처리

- **경기 취소·연기:** match 상태 필드를 `scheduled / postponed / cancelled / finished`로 확장한다. 배트맨은 취소·연기 경기를 적특(전원 적중) 처리하므로, 최종 수집 시 상태를 재확인해 `cancelled`/`postponed` 경기는 부가 데이터 수집과 AI 분석을 건너뛰고 리포트에 "적특 처리 예상"으로 표기한다.
- **마감 시각 변경:** 회차 진행 중 마감이 변경될 수 있으므로 사전 점검(마감−24h)과 최종 수집 시작 시점에 마감 시각을 재조회하고, 달라졌으면 스케줄 잡을 재예약한다.
- **회차 구성 변경:** 경기 교체·수 변경이 감지되면(대진 또는 경기 수 불일치) 회차 파일을 갱신하고 알림을 보낸다.

## 7. AI 분석 파이프라인 (로컬 CLI)

### 7.1 실행 구조

로컬 분석기는 하나의 CLI 명령으로 동작한다: `analyze --round 2026-38`. 실행 시 Git pull로 최신 입력 패키지를 받아 경기별로 Claude를 호출한다. 구현 방식은 **Claude CLI(Claude Code) 헤드리스 호출을 기본으로 확정**한다. 로컬에 이미 설치된 Claude CLI를 `claude -p` 방식으로 스크립트에서 호출하며, 구독 요금 안에서 처리되어 별도 API 비용이 발생하지 않는 것이 확정 이유다. 출력 형식 옵션(JSON)과 프롬프트 말미의 형식 강제를 조합해 7.2절의 구조화 출력을 구현한다. Anthropic Python SDK 직접 호출은 CLI 호출이 병렬 처리·안정성에서 한계를 보일 경우의 대안으로 유지한다. 14경기를 순차 또는 소규모 병렬로 처리하면 회차 전체 분석은 수 분~십수 분 내에 완료된다. 경기당 토큰 규모는 입력 ~3K·출력 ~1.5K 수준으로 구독 사용량에 부담이 없으며, 정확한 CLI 옵션·모델명은 구현 착수 시점에 공식 문서(docs.claude.com)에서 재확인한다.

### 7.2 프롬프트 템플릿과 출력 구조 (초안)

**경기별 분석 프롬프트:**

```
당신은 축구 승무패 분석 전문가다. 아래 JSON은 한 경기의 최신 데이터다(마감 12시간 전 수집).
[경기 JSON 삽입]

다음 구조로 분석하라:
1. 경기 개요 — 리그 맥락, 양 팀의 현재 상황과 동기
2. 전력 비교 — 최근 폼, 홈/원정 성적, 상대 전적의 시사점
3. 변수 — 결장자 영향, 일정 피로도, 로테이션 가능성
4. 시장·대중 해석 — 대중 투표 분포와 해외 시장 확률의 괴리, 시장이 보는 방향과 대중 쏠림
5. 예측 — 승/무/패 확률(합 100%), 예상 스코어 1~2개, 최종 픽과 보조 픽
6. 신뢰도 — 상/중/하 및 그 근거 (데이터 결측이 있으면 명시)

주의: 확률은 시장 배당 내재확률을 기준선으로 삼고, 근거 있는 요인만으로 조정하라.
과신하지 말고 불확실성을 정직하게 기술하라.
```

**출력 이중 구조.** 경기별 분석 출력은 사람이 읽는 서술 마크다운과 기계가 읽는 JSON 블록의 이중 구조로 고정한다. 프롬프트 말미의 형식 요구와 Claude CLI의 JSON 출력 옵션(SDK 대안 시 구조화 출력)으로 아래 형식을 강제하며, 이 JSON이 회차 종합 리포트 생성과 7.4절 적중률 검증의 입력이 된다. 서술만 있고 수치 파싱이 불안정하면 검증 루프 전체가 무너지므로, 형식 위반 시 1회 재요청하는 로직을 둔다.

```json
{
  "probs": {"win": 0.48, "draw": 0.27, "lose": 0.25},
  "pick": "win",
  "backup_pick": "draw",
  "confidence": "중",
  "expected_scores": ["2-1", "1-1"]
}
```

**회차 종합 리포트 프롬프트:**

```
당신은 축구 승무패 분석 편집자다. 아래는 이번 회차 14경기의 경기별 분석 결과 JSON 배열이다.
[경기별 결과 JSON 14개 삽입]

다음 구조로 회차 종합 리포트를 작성하라:
1. 회차 요약 표 — 경기번호, 대진, 최종 픽, 확률, 신뢰도
2. 신뢰도 상위 3경기 — 픽과 근거 요약
3. 이변 후보 — 시장 확률 대비 업셋 가능성이 있는 경기와 이유
4. 투표-시장 괴리 상위 — 대중 투표 분포와 해외 시장 확률의 차이가 큰 경기
5. 종합 코멘트 — 이번 회차의 전반적 난이도와 유의점

수치는 경기별 JSON 값을 그대로 사용하고 새로 추정하지 마라.
취소·연기(적특) 경기는 표에 "적특 예상"으로만 표기하라.
```

### 7.3 산출물

경기별 리포트 14개(`reports/2026-38/M01.md` …)와 회차 종합 리포트 1개를 생성한다. 종합 리포트는 14경기 요약 표(픽/확률/신뢰도), 신뢰도 상위 경기, 이변 후보 경기, 투표-시장 괴리 상위 경기를 담는다. 마크다운 리포트는 **정적 사이트의 회차 페이지(HTML)로 빌드**되어 7.5절 파이프라인으로 배포되며, 화면 디자인은 별첨 문서 「클로드 디자인 요청 가이드」의 토큰과 컴포넌트 규칙을 따른다(경기 카드, 승·무·패 확률 바, 신뢰도 배지, 배당 비교 표). 모든 리포트 하단에 데이터 수집 시각과 참고용 고지를 자동 삽입한다. 선택 기능으로 텔레그램/디스코드 봇 알림을 붙여 완료 즉시 요약을 전송한다.

### 7.4 품질 관리

회차 종료 다음 날 배트맨 결과 페이지에서 14경기 결과를 수집한다(결과 조회 엔드포인트는 4주차에 역공학; API-Football은 무료 플랜 시즌 제한으로 사용 불가). 경기별로 예측 확률 대비 Brier score(3지선다 기준 Σ(pᵢ−oᵢ)², 범위 0~2)와 로그손실을 계산해 6.2절 results 스키마로 기록하고, 회차 단위 적중률·평균 Brier를 누적한다. 취소(적특) 경기는 지표에서 제외한다. 회차 10개가 쌓일 때마다 신뢰도 등급별 실제 적중률과 데이터 항목별 기여도를 리뷰해 프롬프트와 수집 항목을 조정한다. 이 피드백 루프가 있어야 프롬프트 개선과 부가 데이터 가중치 조정이 가능하며, 시스템의 장기 가치가 여기서 나온다.

### 7.5 정적 사이트 빌드·배포 (Cloudflare Pages)

분석 산출물과 회차 데이터는 정적 사이트로 가공해 **Cloudflare Pages에 Git 연동으로 배포**한다. 빌드 도구 없이 순수 HTML을 생성하므로 Pages 설정은 출력 디렉터리(`site/`) 지정만으로 충분하고, `site/`가 커밋·푸시될 때마다 자동 배포된다.

```
site/
├── index.html               # 회차 아카이브 인덱스 (전 회차 누적, 최신이 위)
└── rounds/2026-38/
    ├── index.html           # 회차 페이지 (종합 리포트 / 상태별 내용)
    └── m03.html             # 경기별 상세 페이지 × 14
```

**회차 페이지 라이프사이클.** 하나의 회차 페이지는 세 단계로 갱신되며, 스케줄러가 수집하는 항목이 각 단계에서 계속 반영·누적된다.

| 단계 | 시점 | 페이지 내용 | 생성 주체 |
|---|---|---|---|
| ① 예정 | 회차 감지 직후 | 14경기 리스트(리그·킥오프·배트맨 배당), "분석은 마감 12시간 전 생성" 안내 | 서버(감지 잡이 빌드·커밋) |
| ② 분석 | 마감 −12h 분석 완료 | 종합 리포트 + 경기별 상세 페이지, 확률 바·픽·신뢰도 | 로컬(analyze 후 빌드·푸시) |
| ③ 결과 | 회차 종료 익일 | 실제 결과·적중 여부·Brier 지표를 ②에 덧입힘 | 서버(결과 수집 잡) |

인덱스 페이지는 매 갱신 시 전체 회차를 다시 나열해 과거 회차가 자동으로 아카이브된다. 회차가 쌓이면 인덱스에 회차별 적중률 요약(6.2절 지표)을 함께 노출한다.

**공개 범위.** 4.4절의 "수집 데이터 외부 재배포 금지" 원칙과 충돌하지 않도록, 배포 기본값은 **Cloudflare Access로 본인(허용 이메일)만 접근**하도록 제한한다. 불특정 다수 공개로 전환하는 것은 4.4절의 별도 의사결정(약관 검토) 이후에만 진행한다.

## 8. 기술 스택

| 영역 | 선택 | 비고 |
|---|---|---|
| 언어 | Python 3.12 | 수집·스케줄·CLI 공통 |
| 수집 | httpx (API 직접 호출) + Playwright (폴백) | 4.1절 이중화 |
| 스케줄 | APScheduler (재시작 시 회차 파일에서 잡 복원) | 상대 시각 트리거, 5절 |
| 저장 | JSON 파일 + Git → 추후 SQLite | 이력 자동 관리 |
| AI | Claude CLI 헤드리스 호출 (구독 요금 내, SDK는 대안) | 7.1절에서 확정 |
| 배포 | 정적 HTML → Git push → Cloudflare Pages 자동 배포 | 7.5절, 기본 Access 제한 |
| 알림 | 텔레그램 봇 (선택) | 실패·완료 알림 |
| 서버 | 소형 VPS 1대 (1vCPU/1GB) | Playwright 사용 시 2GB 권장 |

### 8.1 프로젝트 구조

```
ai_football_manager/
├── collector/        # 배트맨 수집(API 직접 호출 + Playwright 폴백), 부가 데이터 수집기
├── scheduler/        # 회차 감지 데몬, APScheduler 잡 정의
├── analyzer/         # 프롬프트 템플릿, Claude CLI 호출, 리포트 생성(md)
├── site_builder/     # 리포트·데이터 → 정적 HTML 빌드 (7.5절 라이프사이클)
├── data/             # rounds/ matches/ reports/ results/ — Git으로 서버↔로컬 공유
├── site/             # 빌드된 정적 페이지 — Cloudflare Pages 배포 대상
├── config/           # config.yaml, teams.yaml
└── tests/            # 픽스처 기반 단위 테스트
```

### 8.2 설정·시크릿 관리

API 키 3종(The Odds API, API-Football, 텔레그램 봇 토큰)은 `.env`로 분리하고 `.gitignore`에 `.env`를 반드시 포함한다. Claude 호출은 로컬 Claude CLI의 로그인 세션을 사용하므로 별도 키가 필요 없다(SDK 대안 채택 시에만 Anthropic 키 추가). 데이터 전달용 저장소가 프라이빗이어도 키는 커밋하지 않는다. 동작 파라미터(감지 시각, 요청 지연, 로컬 폴링 주기, 경로)는 `config/config.yaml`로 관리해 코드 수정 없이 조정한다.

### 8.3 테스트 전략

수집기는 실제 응답을 녹화한 픽스처로 파싱·pydantic 스키마 단위 테스트를 작성해 사이트 개편을 조기에 감지한다. 스케줄러는 KST 시각 계산(마감−12h 산출, 타임존 변환)을 단위 테스트로 고정한다. 분석기는 샘플 입력 패키지 1경기로 프롬프트 변경 시 출력 JSON 형식이 유지되는지 회귀 확인한다.

## 9. 리스크와 대응

사이트 구조 변경이 최대 리스크다. 응답 스키마 검증(6.1절)을 매 수집마다 수행하고, 실패 시 Playwright 폴백과 즉시 알림으로 대응한다. 경기 취소·연기와 마감 시각 변경은 6.3절의 예외 처리 절차를 따른다. 팀명 매핑 누락은 사전 점검(마감-24h) 단계에서 잡는다.

과도한 접근으로 인한 IP 차단 리스크는 저빈도 수집 원칙(4.4절)과 요청 간 지연, 연속 실패 시 즉시 중단·알림으로 관리한다. 로컬 PC가 꺼져 분석이 누락되는 리스크는 마감−11h 감시 잡의 알림과 수동 실행 절차(3절)로 보완한다. 외부 API 무료 쿼터 소진에 대비해 회차당 호출 예산(4.3절)을 지키고 리그 순위처럼 변동이 느린 데이터는 캐시한다.

AI 분석은 Claude CLI 구독 요금 내에서 처리되어 추가 비용이 없으나(회차당 15회 호출 수준), 구독 사용량 한도를 아끼기 위해 입력 패키지를 구조화된 JSON으로 압축한다. 공개 배포로 인한 데이터 재배포 이슈는 7.5절의 Access 제한(기본 비공개)으로 차단한다. 마지막으로, 본 시스템은 구매 판단을 보조하는 참고 도구이며 결과를 보장하지 않는다는 점을 모든 산출물에 명시한다.
