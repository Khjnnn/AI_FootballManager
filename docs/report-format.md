# 분석 리포트 내용 주입 포맷 규칙 (Report Injection Format)

**버전:** v1.0 (2026-07-16, 2026-39 회차 기준 확립)
**목적:** 외부 분석(claude_Football의 Football Match Analyzer V4.0 등)에서 생성된 분석 결과를 본 프로젝트의 `data/reports/{round_key}/`에 주입할 때의 파일 구조·섹션·JSON 스키마 규칙을 정의한다. `site_builder/build.py`가 이 형식을 소비해 정적 페이지를 렌더링하므로, 형식 위반은 사이트 빌드 누락으로 직결된다.

---

## 1. 디렉터리·파일 구조

```
data/reports/{round_key}/          # 예: 2026-39
├── M01.md ~ M14.md               # 경기별 리포트 (14개 필수, 회차 경기 수와 일치)
├── round.md                      # 회차 종합 리포트
└── summary.json                  # 경기별 구조화 데이터 (Mxx.md의 JSON 블록에서 추출)
```

- `round_key`는 `data/rounds/{round_key}.json`과 동일해야 한다 (`{year}-{round_no}`).
- 취소·연기 경기는 리포트에 "적특 처리 예상"으로만 표기하고 JSON `pick`은 생략하지 않는다(빌더 호환).

## 2. 경기별 리포트 `Mxx.md` — 섹션 순서 (고정)

```
# {홈팀} vs {원정팀} — {시즌} {리그} {라운드} 분석 ({날짜} {구장})

## 1. 경기 개요        — 리그 맥락, 순위·승점, 양 팀 현재 상황과 동기
## 2. 전력 비교        — 최근 폼(재개/최신 실전 우선), 득실, H2H 시사점
## 3. 변수             — 결장자, 이적시장 영입/데뷔, 일정 피로, 로테이션
## 4. 시장·대중 해석    — 배트맨 투표 분포 vs 시장/자체 확률 괴리 (쏠림은 신뢰도에만 반영)
## 5. 예측             — 승/무/패 확률(합 100%), 예상 스코어 1~2개, 최종 픽+보조 픽
## 6. 신뢰도: 상/중/하  — 근거, 데이터 결측 명시
## 7. 시나리오 분기     — 표(시나리오/조건/스코어/확률/O·U), 최소 6분기, 합계 100%
## 8. Safety Net       — 3개, 각 옵션에 축 표기(DNB/핸디캡/더블찬스/BTTS/Under)
## 9. ⚡ 업셋 분석      — 조건부(아래 3절), 4항목 고정
**Sources:** — 웹 출처 링크 목록
```json { 구조화 JSON — 아래 4절 } ```
---
*본 분석은 통계적 참고 자료이며 구매 결과를 보장하지 않습니다.*   ← 고지 필수(plan.md 4.4절)
```

### 섹션 작성 규칙

- **§5 확률**: 시장 배당 내재확률(결측 시 자체 추정)을 기준선으로 근거 있는 요인만 조정. 무승부 기저 28% 미만 금지.
- **§7 시나리오**: 확률 합 = 100%, Under 시나리오 합 ≥ 40%, 원정 우세 구도는 "0-1 잠행 원정승" 분기 필수, 저득점 구도는 "0-0" 분기 최소 10%.
- **§8 Safety Net**: Under 계열(U2.5/U3.5/팀골U/BTTS No)은 3개 중 최대 1개. 최소 1개는 픽 반대 방향 헤지.
- **§6 신뢰도 매핑**: 픽 확률 55%+ 및 검증 통과 → `상` / 47~54% 또는 방향 근거 명확 → `중` / 그 외·삼분 구도·핵심 미확인 → `하`.

## 3. 업셋 분석(§9) 주입 조건

다음 중 하나면 §9를 반드시 포함한다:
1. 대중 투표 favorite 60% 이상 경기 (favorite가 홈이든 원정이든)
2. 업셋 신호 2개 이상 확인 경기 (H2H 역상성, 반등임계점, favorite 전력 이탈, 일정 비대칭, 이적생 데뷔 등)
3. 픽과 반대 방향의 경고 태그(H2H 역상성 등)가 있는 경기

**4항목 고정 형식**: 업셋 가능 이유(신호 2~3개) / 업셋 확률 등급(상·중·하) / 반대 논리(2개+) / 최종 판단(픽 유지·조정 여부 + 배팅 시사점).

**등급-확률 일관성**: 등급은 §7 시나리오의 underdog 승리 확률 합과 일치 — 상: 25%+, 중: 15~24%, 하: 8~14%. 업셋 등급 "상"인데 픽 신뢰도 50%+면 재검산.

## 4. 구조화 JSON 블록 스키마 (summary.json의 원천)

각 `Mxx.md` 말미에 ```json 코드펜스로 1개 필수. `site_builder`가 이 블록을 파싱한다.

```json
{
  "match_no": 1,                                  // 정수, 파일명과 일치
  "probs": {"win": 0.32, "draw": 0.32, "lose": 0.36},  // 홈 기준, 합 1.0 (±0.01)
  "pick": "lose",                                 // win | draw | lose
  "backup_pick": "draw",
  "confidence": "하",                             // 상 | 중 | 하
  "expected_scores": ["1-2", "1-1"],              // 1~2개, 홈-원정 표기
  "key_factors": ["...", "..."],                  // 3~4개, 각 1문장
  "data_gaps": ["..."]                            // 결측 항목 명시 (없으면 빈 배열)
}
```

- `summary.json`은 `{"M01": {...}, ..., "M14": {...}}` 형태로, 각 Mxx.md의 JSON 블록을 그대로 추출해 생성한다(수기 작성 금지 — 추출 스크립트 사용):

```bash
python -X utf8 -c "
import json, re
from pathlib import Path
d = Path('data/reports/{round_key}')
out = {}
for f in sorted(d.glob('M*.md')):
    m = re.search(r'\`\`\`json\s*(\{.*?\})\s*\`\`\`', f.read_text(encoding='utf-8'), re.DOTALL)
    out[f.stem] = json.loads(m.group(1))
(d/'summary.json').write_text(json.dumps(out, ensure_ascii=False, indent=2), encoding='utf-8')
"
```

## 5. 회차 종합 리포트 `round.md` — 섹션 순서 (고정)

```
# ⚽ {round_key} 회차 종합 분석 리포트
(회차 성격 요약 1~2문장 — 적특 예상 경기 유무 명시)

## 1. 회차 요약          — 14경기 표: 경기/대진/리그/최종 픽(백업)/승·무·패 확률/신뢰도 + 픽 분포
## 2. 신뢰도 상위 3경기   — 픽 확률 우위 순, 각 근거 2~3줄 + 유의점
## 3. 이변 후보          — 픽과 결과가 어긋날 수 있는 경기와 이유
## 4. 투표-시장 괴리 상위 — 대중 투표 vs 시장(결측 시 자체 분석) 확률 괴리 표
## 5. ⚡ 업셋 세부 분석 요약 — 업셋 대상 경기 표(favorite/방향/등급/신호/판단), 상세는 Mxx §9 참조
## 6. 종합 코멘트        — 회차 난이도·유의점·조합 구성 가이드
---
(참고용 고지 + 수집·분석 시점)
```

- 수치는 경기별 JSON 값을 그대로 사용하고 새로 추정하지 않는다.

## 6. 주입 후 필수 절차

1. `summary.json` 추출·검증: 14경기 존재 + 각 `probs` 합 1.0 (±0.01) assert.
2. 사이트 재빌드: `python -X utf8 -c "from site_builder.build import build_all; build_all()"` → `site/rounds/{round_key}/` 생성 확인.
3. 빌드 페이지에 픽 데이터 임베딩 확인 (`grep pick site/rounds/{round_key}/index.html`).
4. 커밋 스테이징은 `git add data site` (프로젝트 관례 — scheduler/jobs.py `git_push_if_enabled`와 동일 범위).
